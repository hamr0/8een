# M3 — EU interop: evidence log

**Date:** 2026-07-15 · **Machine:** Fedora 44, 8 cores · **Go:** 1.26 ·
**Status: PASSED, with one interop finding.** Rung 1 PASSED (8een reads EU-docType
credentials). Rung 3 PASSED **on the crypto** — 8een verifies a proof generated by the
EU's own longfellow prover — and surfaced a real **circuit-identifier interop gap** that
stops a verbatim EU proof today. Rung 2 (local Multipaz build) was dropped as redundant.

**Exit criterion (PRD §6):** *"The EU app's proofs are format-compatible with
upstream longfellow."* — i.e. 8een verifies a proof it did not itself produce.

Everything through M2 was tested on proofs 8een minted. M3 exists to escape marking
our own homework: to verify a proof made by software we did not write — the EU
age-verification app. This log builds that confidence in three rungs, cheapest first,
each de-risking the next. A rung is not "done" until its check has been *run and shown*.

## The riskiest assumption, split

Format compatibility has two independent halves, and conflating them would let a
failure in one hide behind a pass in the other:

- **(A) Same proof system.** The EU app must prove with the same Longfellow circuits
  8een verifies against. If the circuit parameters differ, nothing else matters.
- **(B) No hidden mDL assumption in 8een.** 8een's own reader must not be wired to the
  ISO 18013-5 mDL docType/namespace. The EU credential is `eu.europa.ec.av.1`. If 8een
  silently assumes mDL, a real EU proof is refused for a reason unrelated to the proof
  — and that is *our* bug, indistinguishable at a glance from the app being incompatible.

Rung 1 settles (A) by measurement and (B) by running the whole §7.1 matrix under the
EU docType. Rungs 2–3 raise the confidence from "8een handles EU-shaped credentials"
to "8een verifies a genuine EU-app proof."

## Rung 1 — the local half (no emulator) — **PASSED**

### 1.1 (A) The EU app proves with circuits 8een already pins — measured, byte-for-byte

The EU app's Longfellow circuits ship in `eudi-lib-android-wallet-core`'s AAR
(`wallet-core/src/main/assets/circuits/longfellow-libzk-v1/`); the EU verifier's ship
in `av-dc-api-backend`'s vendored `multipaz-longfellow/src/commonMain/circuits/`. Both
sets were compared against 8een's pinned manifest (`src/circuits.manifest.json`, 17
circuits) by sha256:

| Source | circuits | vs 8een's pinned sha256 |
|---|---|---|
| EU app (prover), wallet-core AAR | 4 (the `7_*` family) | **4/4 byte-identical** |
| EU verifier, av-dc-api-backend | 8 (`6_*` and `7_*`) | **8/8 byte-identical** |

**12/12 byte-identical, 0 differing.** The EU did not fork the crypto — they ship
upstream Longfellow circuits unmodified, and every one is in the set 8een provisions.
Half (A) holds: the proof systems are the same.

### 1.2 (B) 8een sends the child no docType — read from source

`src/service.js` (the `verify` path) posts the child exactly two fields:
`Transcript` and `ZKDeviceResponseCBOR`. **No docType, no namespace.** The credential's
docType lives inside the opaque CBOR blob, which 8een never inspects. And
`findClaim` (`src/verdict.js`) scans *every* namespace for the required element
identifier, pinning none. So on inspection 8een has no mDL assumption to trip on —
but inspection is not evidence, so:

### 1.3 (B) The §7.1 matrix, re-run under the EU docType — the falsifiable check

`tools/mkfixture` gained `-doctype` / `-namespace` flags (default ISO mDL). The
element identifier stays `age_over_18` — confirmed to be the EU AV attribute name too
(`av-dc-api-backend/multipaz-doctypes/.../AgeVerification.kt:71-75`), so no flag needed.

The full fixture matrix was minted under `eu.europa.ec.av.1` and run through 8een's
**public API** (`VerifierService` + `Verifier`), asserting each row against the same
verdict its mDL twin gets (`test/integration.test.js`, "M3 rung 1"):

| Row | Verdict | Layer proven docType-agnostic |
|---|---|---|
| valid | `ok:true, over_threshold:true, verified` | accept + claim found under EU namespace |
| underage | `ok:true, over_threshold:false, claim_false` | the `ok`/`over_threshold` split |
| untrusted-issuer | `ok:true, over_threshold:false, issuer_untrusted` | chain validation |
| tampered | `ok:true, over_threshold:false, zk_proof_invalid` | the ZK math |

**Run: 4/4 pass (~105 s). Full suite: 24/24, 0 skipped, no regression to the mDL matrix.**

**Non-vacuity.** Two reject rows, each failing at a *different* layer (chain vs ZK), so
a pass cannot come from a verifier that accepts everything. And the accept rows keep
`ok` separate from `over_threshold`, so a pass cannot come from one that collapses them.

**Confound ruled out — did the test actually exercise the variable?** The minted EU
fixture's `ZKDeviceResponseCBOR` was decoded and checked: it contains the literal
`eu.europa.ec.av.1` and **not** `org.iso.18013`; the default mDL fixture is the exact
inverse. So the flags are not silently no-ops — the matrix genuinely ran under the EU
strings. (A missing version of this check is the classic trap: a test that "passes"
against a fixture that never actually changed.)

**Corroboration that the crypto layer agrees.** `mkfixture` refuses to emit a fixture
it has not itself put through longfellow's real prover *and* verifier
(`assertFixtureVerifies`). The EU-docType set minting at all means longfellow accepts
the EU-docType accept-fixtures and rejects the negatives — independently of 8een's
classification layer.

### 1.4 What rung 1 does and does not establish

**Establishes:** the necessary local precondition — 8een reads an EU-docType,
EU-namespace credential exactly as an mDL one, across accept and both reject layers,
and the proof system is provably identical. A real EU-app proof failing for a
docType/namespace reason is now ruled out.

**Does not establish:** that a *genuine EU-app proof* verifies. Every proof in rung 1
was still minted by 8een's own tooling under a CA 8een controls. The remaining gap —
a real device signature, a real EU issuer chain, a real session transcript — is what
rungs 2–3 close.

## Rung 2 — local differential oracle — **DROPPED (folded into rung 3)**

The original plan was to build the EU's own ZK verifier (`av-dc-api-backend`, vendored
Multipaz) locally and cross-check our fixtures against it. Scoping killed it:

- **Format mismatch.** 8een's fixtures are longfellow's reference verifier-service wire
  format (`{Transcript, ZKDeviceResponseCBOR}`). Multipaz only verifies through its HTTP
  session API (`dcBegin`/`dcGetData`/`annexA`), which builds its *own* session transcript;
  our proof is bound to *our* transcript, so it cannot validate against a Multipaz session
  even after reformatting. Not a small translation — a reverse-engineering exercise.
- **Redundant crypto core.** `multipaz-longfellow` bundles the *same* Google longfellow-zk
  library 8een's reference verifier links (a prebuilt `libzkp.so` ships in the JVM
  resources). At the cryptographic core the two agree by construction; the differential
  would only exercise the wrapper (CBOR parse, chain validation, key extraction).
- **Cost.** A 32-module Kotlin/JVM Gradle build for that narrow, wrapper-only signal.

**The independent-oracle value is obtained for free in rung 3 instead:** the EU's verifier
is *deployed and live* (reached with `curl` in `EU-STACK-AUDIT.md` §5). A real EU-app proof
can be verified against the live EU verifier AND 8een — a stronger differential than a
local build (real proof, real independent verifier, natively EU-shaped, no translation).

## Rung 3 — a real EU-produced proof, verified by 8een — **PASSED (crypto), with a finding**

### 3.0 The emulator wall (why the path changed)

The first plan was to capture a proof off the EU app on an Android emulator. That is a
hard dead end **on this host**: a freshly-installed, correctly-configured Android
emulator (`36.6.11`, KVM confirmed usable) **SIGSEGVs on boot** — a null-pointer jump in
`qemu-system-x86_64-headless` right after memory/display setup, reproducible across every
GPU mode (`swiftshader_indirect`/`guest`/`off`) and with ASLR disabled, and identical on
the only other emulator build Google indexes. Root cause from the coredump: the emulator
binary vs this host's bleeding-edge **kernel 7.1.3 / Fedora 44**. The *windowed* qemu
survived the segfault but never brought up its adb bridge. Not papered over — recorded so
nobody re-treads it. (The system image never mattered: the crash is pre-guest.)

### 3.1 The path that worked — run the EU's OWN prover on the JVM

Instead of a phone, run the EU verifier's own ZK library directly. `av-dc-api-backend`
(the EU's deployed ZK verifier, vendored Multipaz) contains `multipaz-longfellow`, which
ships a **prebuilt JVM `libzkp.so`** and a runnable `SystemTest` that calls
`LongfellowZkSystem.generateProof(...)`. Built it (JDK 21 for Gradle 8.14; the host's
JDK 25 is too new for Gradle to run on; `ANDROID_HOME` = the SDK installed above; a local
`0.99.0` tag to satisfy the git-derived version) and confirmed the EU prover runs here:
`verify done: ok`, `Verification Code: 0`.

A one-shot emitter (`EmitProofTest`, mirroring `SystemTest`) then generates a proof over
the EU's own mdoc test vector and writes it in 8een's wire format
(`{Transcript, ZKDeviceResponseCBOR}`). This is a proof produced by software 8een did not
write — the whole point of M3.

### 3.2 Wire-format interop — 8een decodes the EU envelope byte-for-byte

8een's exact reference-verifier structs decoded Multipaz's `DeviceResponse` with no
adaptation: `Version="1.1"`, `Status=0`, `DocType="org.iso.18013.5.1.mDL"`,
namespace `org.iso.18013.5.1`, proof 323 836 B. The CBOR envelope the EU prover emits **is**
the envelope 8een's verifier consumes.

### 3.3 The chain — the EU test issuer, on the real clock

The proof's `MsoX5chain` carries one cert: DS `CN=OWF Multipaz TEST DS`, issued by
`CN=OWF Multipaz TEST IACA`, valid **2025-06-29 … 2026-09-27** — *not* expired on the real
clock (unlike upstream `post1.json`). Fed with an unrelated trust list, 8een refuses at
the chain (`issuer_untrusted`, "unknown authority") — proving it parsed and reached chain
validation. Trusting the EU DS cert, the chain validates and 8een proceeds to the ZK layer.

### 3.4 THE FINDING — circuit-identifier convention mismatch

A **verbatim** EU proof does **not** verify today. Its `ZkSystemId` is the composite
`longfellow-libzk-v1_6_1_4096_2945_137e5a75…` (Multipaz's `"${systemName}_${circuitFileName}"`,
`LongfellowZkSystem.kt:123`). 8een's reference verifier keys circuits by their **bare
`circuit_id`** (the 64-hex hash) and *ignores* any circuit file whose id ≠ filename
(`circuits.go:75`), so `GetCircuitByName` cannot resolve the composite label:

```
real EU proof  →  {"ok":false,"over_threshold":null,"reason":"circuit_unavailable",
                   "detail":"invalid circuit id: longfellow-libzk-v1_6_1_4096_2945_137e5a75…"}
```

**The invariant holds exactly where it must:** an unresolvable circuit yields
`ok:false ⇒ over_threshold:null` — *"I cannot verify,"* never a false "underage." Two
longfellow verifier implementations (Google's reference, which 8een wraps; the EU's
Multipaz) disagree on the on-wire circuit-id convention, over byte-identical circuits.

### 3.5 The crypto DOES interoperate — the gap is the label, nothing else

Emitting the same proof with `ZkSystemId` set to the bare hash 8een expects (a one-line
spec-id change; the proof bytes are otherwise the EU prover's) isolates the crypto from the
label:

```
EU proof, bare-hash id  →  {"ok":true,"over_threshold":true,"reason":"verified","detail":"age_over_18"}
tampered (one byte)     →  {"ok":true,"over_threshold":false,"reason":"zk_proof_invalid"}
```

**8een verifies a genuine EU-prover-generated ZK proof** (accept), and the accept is
non-vacuous (a one-byte flip flips it to `zk_proof_invalid`). So the ZK circuits, the
proof math, the CBOR envelope, and the cert chain all interoperate. The **only** thing
between 8een and a verbatim EU proof is the circuit-id string convention (§3.4).

### 3.6 Honest scope

The prover exercised is `av-dc-api-backend`'s `multipaz-longfellow` — the EU's own ZK
verifier library, the same longfellow family the app links via `eudi-lib-android-wallet-core`.
The exact on-phone `ZkSystemId` the shipping app emits (through wallet-core's
`LongfellowZkSystemRepository`, a further wrapper) was **not** pinned here; it may match
this convention or differ again. What is proven: a proof from genuine EU longfellow code
verifies under 8een's crypto, and the reference-verifier circuit-id convention differs
from the EU/Multipaz one. The live-EU-verifier cross-check was not needed — the JVM prover
is a stronger, more direct interop test than a black-box round-trip.

### 3.7 The open decision

Should 8een resolve circuits by the EU/Multipaz composite `ZkSystemId` (e.g. match on the
trailing hash segment), so a verbatim EU proof verifies? That is a labeling/resolution
change, **not** longfellow crypto (NO-GO #8 untouched) — but it edits the vendored
reference verifier's id handling, so it is an owner decision, recorded here, not made here.
It is also a candidate finding to report upstream: the reference verifier and the EU's
Multipaz stack do not agree on the circuit-id wire convention.
