**Exit criterion (PRD ┬з6):** *"The EU app's proofs are format-compatible with
upstream longfellow."* тАФ i.e. 8een verifies a proof it did not itself produce.

Everything through M2 was tested on proofs 8een minted. M3 exists to escape marking
our own homework: to verify a proof made by software we did not write тАФ the EU
age-verification app. This log builds that confidence in three rungs, cheapest first,
each de-risking the next. A rung is not "done" until its check has been *run and shown*.

## The riskiest assumption, split

Format compatibility has two independent halves, and conflating them would let a
failure in one hide behind a pass in the other:

- **(A) Same proof system.** The EU app must prove with the same Longfellow circuits
  8een verifies against. If the circuit parameters differ, nothing else matters.
- **(B) No hidden mDL assumption in 8een.** 8een's own reader must not be wired to the
  ISO 18013-5 mDL docType/namespace. The EU credential is `eu.europa.ec.av.1`. If 8een
  silently assumes mDL, a real EU proof is refused for a reason unrelated to the proof
  тАФ and that is *our* bug, indistinguishable at a glance from the app being incompatible.

Rung 1 settles (A) by measurement and (B) by running the whole ┬з7.1 matrix under the
EU docType. Rungs 2тАУ3 raise the confidence from "8een handles EU-shaped credentials"
to "8een verifies a genuine EU-app proof."

### 1.2 (B) 8een sends the child no docType тАФ read from source

`findClaim` (`src/verdict.js`) scans *every* namespace for the required element

### 1.3 (B) The ┬з7.1 matrix, re-run under the EU docType тАФ the falsifiable check

### 1.4 What rung 1 does and does not establish

**Establishes:** the necessary local precondition тАФ 8een reads an EU-docType,

## Rung 2 тАФ local differential oracle тАФ **DROPPED (folded into rung 3)**

The original plan was to build the EU's own ZK verifier (`av-dc-api-backend`, vendored
Multipaz) locally and cross-check our fixtures against it. Scoping killed it:

- **Format mismatch.** 8een's fixtures are longfellow's reference verifier-service wire
  format (`{Transcript, ZKDeviceResponseCBOR}`). Multipaz only verifies through its HTTP
  session API (`dcBegin`/`dcGetData`/`annexA`), which builds its *own* session transcript;
  our proof is bound to *our* transcript, so it cannot validate against a Multipaz session
  even after reformatting. Not a small translation тАФ a reverse-engineering exercise.
- **Redundant crypto core.** `multipaz-longfellow` bundles the *same* Google longfellow-zk
  library 8een's reference verifier links (a prebuilt `libzkp.so` ships in the JVM
  resources). At the cryptographic core the two agree by construction; the differential
  would only exercise the wrapper (CBOR parse, chain validation, key extraction).
- **Cost.** A 32-module Kotlin/JVM Gradle build for that narrow, wrapper-only signal.

**The independent-oracle value is obtained for free in rung 3 instead:** the EU's verifier
is *deployed and live* (reached with `curl` in `EU-STACK-AUDIT.md` ┬з5). A real EU-app proof
can be verified against the live EU verifier AND 8een тАФ a stronger differential than a
local build (real proof, real independent verifier, natively EU-shaped, no translation).

## Rung 3 тАФ a real EU-produced proof, verified by 8een тАФ **PASSED (crypto), with a finding**

### 3.0 The emulator wall (why the path changed)

The first plan was to capture a proof off the EU app on an Android emulator. That is a
hard dead end **on this host**: a freshly-installed, correctly-configured Android
emulator (`36.6.11`, KVM confirmed usable) **SIGSEGVs on boot** тАФ a null-pointer jump in
`qemu-system-x86_64-headless` right after memory/display setup, reproducible across every
GPU mode (`swiftshader_indirect`/`guest`/`off`) and with ASLR disabled, and identical on
the only other emulator build Google indexes. Root cause from the coredump: the emulator
binary vs this host's bleeding-edge **kernel 7.1.3 / Fedora 44**. The *windowed* qemu
survived the segfault but never brought up its adb bridge. Not papered over тАФ recorded so
nobody re-treads it. (The system image never mattered: the crash is pre-guest.)

### 3.1 The path that worked тАФ run the EU's OWN prover on the JVM

Instead of a phone, run the EU verifier's own ZK library directly. `av-dc-api-backend`
(the EU's deployed ZK verifier, vendored Multipaz) contains `multipaz-longfellow`, which
ships a **prebuilt JVM `libzkp.so`** and a runnable `SystemTest` that calls
`LongfellowZkSystem.generateProof(...)`. Built it (JDK 21 for Gradle 8.14; the host's
JDK 25 is too new for Gradle to run on; `ANDROID_HOME` = the SDK installed above; a local
`0.99.0` tag to satisfy the git-derived version) and confirmed the EU prover runs here:
`verify done: ok`, `Verification Code: 0`.

A one-shot emitter (`EmitProofTest`, mirroring `SystemTest`) then generates a proof over
the EU's own mdoc test vector and writes it in 8een's wire format
(`{Transcript, ZKDeviceResponseCBOR}`). This is a proof produced by software 8een did not
write тАФ the whole point of M3.

### 3.2 Wire-format interop тАФ 8een decodes the EU envelope byte-for-byte

8een's exact reference-verifier structs decoded Multipaz's `DeviceResponse` with no
adaptation: `Version="1.1"`, `Status=0`, `DocType="org.iso.18013.5.1.mDL"`,
namespace `org.iso.18013.5.1`, proof 323 836 B. The CBOR envelope the EU prover emits **is**
the envelope 8een's verifier consumes.

### 3.3 The chain тАФ the EU test issuer, on the real clock

The proof's `MsoX5chain` carries one cert: DS `CN=OWF Multipaz TEST DS`, issued by
`CN=OWF Multipaz TEST IACA`, valid **2025-06-29 тАж 2026-09-27** тАФ *not* expired on the real
clock (unlike upstream `post1.json`). Fed with an unrelated trust list, 8een refuses at
the chain (`issuer_untrusted`, "unknown authority") тАФ proving it parsed and reached chain
validation. Trusting the EU DS cert, the chain validates and 8een proceeds to the ZK layer.

### 3.4 THE FINDING тАФ circuit-identifier convention mismatch

A **verbatim** EU proof does **not** verify today. Its `ZkSystemId` is the composite
`longfellow-libzk-v1_6_1_4096_2945_137e5a75тАж` (Multipaz's `"${systemName}_${circuitFileName}"`,
`LongfellowZkSystem.kt:123`). 8een's reference verifier keys circuits by their **bare
`circuit_id`** (the 64-hex hash) and *ignores* any circuit file whose id тЙа filename
(`circuits.go:75`), so `GetCircuitByName` cannot resolve the composite label:

```
real EU proof  тЖТ  {"ok":false,"over_threshold":null,"reason":"circuit_unavailable",
                   "detail":"invalid circuit id: longfellow-libzk-v1_6_1_4096_2945_137e5a75тАж"}
```

**The invariant holds exactly where it must:** an unresolvable circuit yields
`ok:false тЗТ over_threshold:null` тАФ *"I cannot verify,"* never a false "underage." Two
longfellow verifier implementations (Google's reference, which 8een wraps; the EU's
Multipaz) disagree on the on-wire circuit-id convention, over byte-identical circuits.

### 3.5 The crypto DOES interoperate тАФ the gap is the label, nothing else

Emitting the same proof with `ZkSystemId` set to the bare hash 8een expects (a one-line
spec-id change; the proof bytes are otherwise the EU prover's) isolates the crypto from the
label:

```
EU proof, bare-hash id  тЖТ  {"ok":true,"over_threshold":true,"reason":"verified","detail":"age_over_18"}
tampered (one byte)     тЖТ  {"ok":true,"over_threshold":false,"reason":"zk_proof_invalid"}
```

**8een verifies a genuine EU-prover-generated ZK proof** (accept), and the accept is
non-vacuous (a one-byte flip flips it to `zk_proof_invalid`). So the ZK circuits, the
proof math, the CBOR envelope, and the cert chain all interoperate. The **only** thing
between 8een and a verbatim EU proof is the circuit-id string convention (┬з3.4).

### 3.6 Honest scope

The prover exercised is `av-dc-api-backend`'s `multipaz-longfellow` тАФ the EU's own ZK
verifier library, the same longfellow family the app links via `eudi-lib-android-wallet-core`.
The exact on-phone `ZkSystemId` the shipping app emits (through wallet-core's
`LongfellowZkSystemRepository`, a further wrapper) was **not** pinned here; it may match
this convention or differ again. What is proven: a proof from genuine EU longfellow code
verifies under 8een's crypto, and the reference-verifier circuit-id convention differs
from the EU/Multipaz one. The live-EU-verifier cross-check was not needed тАФ the JVM prover
is a stronger, more direct interop test than a black-box round-trip.

### 3.7 The decision тАФ RESOLVED: the circuit-id bridge (owner-approved 2026-07-15)

The open question was whether 8een should resolve circuits by the EU/Multipaz composite
`ZkSystemId` so a verbatim EU proof verifies. The owner approved it (Option 1 of three:
approve as a named patch / leave as documented finding / attempt a non-vendored fix). It
is a labeling/resolution change, **not** longfellow crypto (NO-GO #8 untouched), captured
as a tracked, reproducible patch rather than a silent edit to the throwaway clone:

```
poc/patches/0002-eu-circuit-id-compat.patch
```

**What it does.** Adds `NormalizeCircuitID(id)` to `circuits.go` and applies it once at the
top of `VerifyProofRequest` (`proofs.go`), so both the circuit-map lookup *and* the C
library's `find_zk_spec` receive the bare id. The rule: if the full `ZkSystemId` is a known
circuit, use it; else if the segment after the last `_` is a known circuit, use that; else
return the id unchanged.

**Why it cannot select a wrong or weaker circuit.** `LoadCircuits` only admits a circuit
whose content hash equals its map key (longfellow's own `circuit_id()`, `circuits.go:75`),
so every key is a cryptographically-verified circuit id. Normalizing a composite id to its
trailing segment therefore resolves to the *same* verified circuit or to nothing тАФ never a
different one. An id whose trailing segment is not a known circuit passes through unchanged,
so the normal `invalid circuit id` тЖТ `circuit_unavailable` (`ok:false тЗТ over_threshold:null`)
still fires. The prefix is cosmetic; the trust boundary and the ZK math are untouched, and
`age_over_18`-must-be-present-and-true remains a second gate behind circuit selection.

**Verified end-to-end (real EU-prover proof, patched server rebuilt, real clock):**

| Case | Input | Verdict |
|---|---|---|
| baseline (pre-patch) | verbatim composite + trusted DS | `ok:false, over_threshold:null, circuit_unavailable` |
| **bridge** | **verbatim composite + trusted DS** | **`ok:true, over_threshold:true, verified, age_over_18`** |
| trust still enforced | verbatim composite + *untrusted* list | `ok:true, over_threshold:false, issuer_untrusted` |
| non-vacuous | tampered (one byte) + trusted DS | `ok:true, over_threshold:false, zk_proof_invalid` (code 5) |

The bridge converts the *exact* baseline failure (`circuit_unavailable` on the verbatim,
unmodified EU proof) into `verified` тАФ no longer the one-line emitter relabel of ┬з3.5, but
the proof bytes as the EU prover emitted them. Trust and non-vacuity are unchanged by the
bridge (rows 3тАУ4), and the invariant holds throughout. **Full regression after the rebuild:
`npm run test:integration` тЖТ 24/24 pass, 0 skipped.**

**Reproduce.** Apply the patch in the clone and rebuild the reference server:
`git -C poc/longfellow-zk apply poc/patches/0002-eu-circuit-id-compat.patch` then
`CGO_ENABLED=1 go build -o server .` in `reference/verifier-service/server`. Note the
committed integration suite does **not** depend on this patch тАФ its fixtures are 8een-minted
with bare-hash circuit ids, so it passes on a pristine (0001-only) server too. The patch is
required only to reproduce the *verbatim-EU-proof* result above, whose EU-prover proof and
test-CA material are not committed (PRD ┬з10; regenerate via the rung-3 recipe in ┬з3.1).
